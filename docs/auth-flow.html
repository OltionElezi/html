🔒 Authentication & Security

The system uses three authentication methods. The primary flow is JWT via AWS Cognito for human users. API Keys are used for machine-to-machine calls. Basic Auth is available for the operations endpoint.

🔐 Authentication Methods Summary

MethodWho uses itHowRoles granted
JWT (Cognito) Human users via Angular dashboard Authorization: Bearer <jwt> header — added by Angular AuthInterceptor From cognito:groups JWT claim
API Key External services, integrations API key in request header, processed by APIKeyAuthFilter ROLE_admin, ROLE_web, ROLE_currency_config
Basic Auth Operations team (/ops* endpoints) HTTP Basic authentication, in-memory user ROLE_admin

🔄 JWT Login Flow (Step by Step)

This is the complete flow from user clicking "Login" to receiving an authenticated API response:

Angular (Amplify)
Spring Boot API
Step 1 — User enters email + password on Amplify login UI
Browser ──────────────────────────────────────────> Cognito
POST /oauth2/token {email, password}
Step 2 — Cognito validates credentials, returns JWT tokens
Cognito ─────────────────────────────────────────> Browser
Returns: { accessToken, idToken, refreshToken }
Step 3 — Amplify stores tokens in memory / localStorage
Amplify stores: accessToken, idToken, refreshToken
Step 4 — User navigates, Angular makes an HTTP request
AuthInterceptor calls Auth.currentSession()
Gets accessToken from Amplify session
Step 5 — Interceptor adds JWT to request header
Angular ──────────────────────────────────────────> Spring Boot
GET /api/... with: Authorization: Bearer <accessToken>
Step 6 — Spring Boot validates JWT (online — fetches JWKS from Cognito)
Spring ─────────────────────────────────────────> Cognito
GET /.well-known/jwks.json (first time only, then cached)
Step 7 — JWT validated. Extracts cognito:groups → Spring ROLE_*
JWT claim cognito:groups: ["admin"]
→ Spring authority: ROLE_admin
Step 8 — Authorization check passes, business logic executes, response returned
Spring Boot ──────────────────────────────────> Angular
HTTP 200 OK { data: ... }

📝 JWT Validation Details

# application.yml — Backend JWT configuration
spring.security.oauth2.resourceserver.jwt:
  issuer-uri: https://cognito-idp.eu-central-1.amazonaws.com/eu-central-1_8Db4kY02L

# JWKS endpoint auto-derived from issuer-uri:
# https://cognito-idp.eu-central-1.amazonaws.com/eu-central-1_8Db4kY02L
#   /.well-known/jwks.json
// WebApplicationSecurityConfig.java — JWT to Spring authorities mapping
JwtGrantedAuthoritiesConverter converter = new JwtGrantedAuthoritiesConverter();
converter.setAuthoritiesClaimName("cognito:groups");  // which claim to read
converter.setAuthorityPrefix("ROLE_");               // prefix added to each group

// Example JWT payload (decoded):
// {
//   "sub": "user-uuid",
//   "cognito:groups": ["admin", "reservationmanager"],
//   "email": "user@example.com",
//   "exp": 1234567890
// }
// → Spring authorities: [ROLE_admin, ROLE_reservationmanager]

🔑 API Key Authentication

The APIKeyAuthFilter is a Spring Security pre-authentication filter. It runs before JWT validation and grants access if a valid API key is found in the request header.

// How API key auth works:
1. Request arrives with API key in header
2. APIKeyAuthFilter reads the header value
3. Looks up key in application.yml api-keys map
4. If found: creates Authentication with mapped role
5. Sets principal as static string "API_KEY"
6. Spring Security proceeds with granted authorities

👤 Basic Auth — Operations Endpoint

An in-memory Spring Security user is defined for operations access to /ops* endpoints.

// application.yml
fmtg.ops:
  username: fmtg_manager
  password: $2y$12$mu0TU...  # BCrypt-hashed

// WebApplicationSecurityConfig.java
auth.inMemoryAuthentication()
    .withUser("fmtg_manager")
    .password(hashedPassword)
    .authorities("ROLE_admin");

// Usage:
// Authorization: Basic base64(fmtg_manager:password)
// Only works for /ops* endpoints

🏆 Roles & Permissions

RoleSourceAccess Level
ROLE_admin Cognito group admin, Basic Auth, API key Full access to all endpoints. Manages users, roles, properties, seasons, content, experiences, discount codes, rate codes. Access to /ops*.
ROLE_reservationmanager Cognito group reservationmanager Reservation management, offers, payments, confirmations, tickets. Cannot access user/role/property management.
ROLE_cms Cognito group cms Email template management (view & edit). Limited to CMS content.
ROLE_web API key only Web/frontend public API access for integrations.
ROLE_currency_config API key only Currency configuration updates.

🛣 Frontend FeatureGuard Flow

On the frontend, every protected route is guarded by FeatureGuard which performs two checks:

FeatureGuard.canActivate() flow:
1. Read route data: { roles: ['admin'], feature: 'user_management' }
2. Get current user from AuthService (Amplify session + decoded JWT)
3. Check: does user have any of the required roles?
NO → redirect to /access-denied
YES → continue to check 4
4. Check: does user's entity have the required feature flag enabled?
NO → redirect to /access-denied
YES → allow navigation, component loads

ⓘ Feature flags are per-entity (hotel property)

Each hotel entity can have different features enabled. For example, one property might have asm_tickets enabled but not custom_offer_creation. This allows per-property capability configuration without changing user roles.
